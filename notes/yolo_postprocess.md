# 话题:YoloPostprocess(NMS/IoU/box解码)

记录格式约定:核心一句话 / 要点N / 注意事项 / 代码位置参考

## 核心一句话
这个类做的是"NPU吐出的原始浮点输出→人能看懂的检测框列表"这一步,核心是box坐标解码+按类别NMS去重叠框;最容易被深挖的是预处理用整图拉伸而不是letterbox这个设计选择,以及NMS为什么只对同类别框互相抑制。

## 要点1:整图拉伸缩放,不是letterbox(待核实点,面试前要确认)
`sx = orig_w / cfg_.model_input_size`、`sy = orig_h / cfg_.model_input_size`(line 24-25)分开算,说明预处理是把原始帧(640×480)直接拉伸塞进640×640模型输入,不保持长宽比。很多YOLO实现默认用letterbox(等比缩放+灰边填充,避免画面形变),这里没有用。

**待核实**:训练/INT8量化转换阶段(`tools/convert_int8.py`或更早的ONNX导出环节)的预处理是不是也用同样的直接拉伸。
- 如果训练侧也是直接拉伸→两边一致,是"图简单且经过验证没问题"的正常设计选择,没有故事可挖,一句话带过即可。
- 如果训练侧用了letterbox、推理侧却用拉伸→预处理不一致,模型看到的几何关系跟训练时不一样,是个没写进`BUGS.md`的潜在精度问题,比单纯"设计选择"更有挖掘价值,但**目前还没验证,面试时不要在没确认的情况下主动断言这是个bug**。
- 这条待面试前去读相关脚本确认后再定最终表述。

## 要点2:cls不能重复sigmoid(BUG-009同一次排查里的小bug,一句话带过)
跟`infer_thread.md`要点3"双输出独立量化scale"是同一次排查(BUG-009)发现的两层问题里的第二层:YOLOv8导出的ONNX图分类分支自带`Sigmoid`节点,`process()`里直接用`cls[c*n_anchors+a]`当概率用(line 31注释里特别标注"已经过Sigmoid,不能再套一次"),如果手动再做一次sigmoid会把概率值压缩失真。
- 不单独展开,提一句"同一次排查顺带修的小问题"即可,详细故事在`infer_thread.md`要点3。

## 要点3:NMS只对同类别框互相抑制
```cpp
if (!suppressed[j] &&
    dets[i].class_id == dets[j].class_id &&     // 只比较同类别
    iou(dets[i], dets[j]) > cfg_.iou_threshold)
    suppressed[j] = true;
```
按score从高到低排序,从最高分开始依次保留,把同类别且IoU超过阈值的后续框标记为重复框丢弃;不同类别框允许重叠——比如人骑车场景,人和车的框天然大面积重叠,但属于不同物体,不应该互相抑制。
- 追问:这样会不会漏检?——会,如果同类别两个真实物体离得很近、框重叠度超过阈值,会被错误合并成一个,这是NMS算法本身的已知局限,不是这个项目独有的实现bug。
- 复杂度O(n²),n是过滤阈值后剩下的候选框数(一般几十个量级),够用,不需要优化成更复杂的实现。

## 要点4:argmax找最高分类别
逐个类别线性扫描(`for c in 0..79`)找最高分,O(80)/anchor,写法标准没有设计权衡空间。
- 如果被问到,提一句复杂度即可,不值得展开。

## 注意事项
1. 优先准备要点3,是最常被问到、也最容易讲清楚的部分。
2. 要点1是这个话题里唯一有"潜在问题"色彩的点,面试前必须先去确认训练侧预处理逻辑,不要带着没验证的猜测去答。
3. 要点2不要重复展开`infer_thread.md`已经讲过的BUG-009内容,提一句交叉引用即可。
