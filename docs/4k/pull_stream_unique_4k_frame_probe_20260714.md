# 独立地址 4K 实际帧完整性探针（2026-07-14）

## 测试条件

- 输入文件：`runs/input_videos/4k/VID_20260712_131410.mp4`
- 输入文件参数：3840×2160、H.264、yuvj420p、30 FPS
- 板端源类型：`pull_stream`
- 板端源地址：`rtmp://192.168.0.168/live/testsrc2`
- 独立输出地址：`rtmp://192.168.0.168/live/pull_4k_verify_20260714`
- 配置文件：`runs/testsrc2/config_pull_stream_unique_4k_20260714.json`
- MPP 输出配置：3840×2160、30 FPS、6000 kbps

## 验证结果

1. 板端日志显示解码器打开 `3840x2160, yuvj420p`，没有出现配置尺寸不匹配丢帧。
2. SRS API 显示独立输出为活跃的 H.264 Main、3840×2160。
3. 从独立 RTMP 地址抓取的 `/tmp/pull_4k_verify_20260714.flv` 经 ffprobe 确认为：
   `h264 / Main / 3840×2160 / yuv420p / 30 FPS / level 5.1`。
4. 抓取文件抽出的实际视频帧 `/tmp/pull_4k_verify_20260714.png` 为完整 16:9 画面，未观察到
   仅显示左上角或局部区域的裁切现象。
5. 测试源结束后，板端进入正常的拉流重连等待；测试进程已停止。

## 结论

当前 4K `pull_stream` 配置和编码推流代码在独立输出地址下能够生成完整 3840×2160 视频，
本次证据不支持“配置尺寸导致编码只输出局部画面”或“MPP 输入 stride 导致固定裁切”的判断。

此前复用了 V4L2 测试地址；该地址曾经发布过 1280×720 流。若播放器没有重新建立连接，或
SRS/播放器仍沿用旧的 SPS/分辨率状态，就可能出现画面按旧尺寸解释、看起来只显示局部的现象。
验证时应使用新的推流地址，或停止旧流后让播放器重新连接并重新获取 SPS/PPS。

若独立地址仍能复现局部画面，下一步才需要针对播放器解码日志、SPS/PPS 和 MPP buffer/stride
做代码级定位；目前不应先改动 4K 尺寸传递链路。
