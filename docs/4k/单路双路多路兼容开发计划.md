# 单路、双路与多路兼容开发计划

## 1. 目的

将 AlertGateway 当前“一个进程、一条流水线”的实现，逐步扩展为：

- 旧单路配置继续可用；
- 新配置可以在同一个进程中运行两路；
- 后续通过增加配置项扩展到更多路，不复制 `main.cpp` 的整套启动逻辑；
- 单路、双路和多路都能分别确认输入、推理、编码、输出和检测数据归属。

本计划只定义开发路线，当前不直接改动 C++ 实现。现有 `run.sh` 仍通过注释切换单路、双路和未来多路测试脚本。

## 2. 当前实现与主要问题

当前 `src/main.cpp` 直接在函数栈上创建一套完整流水线：

```text
IVideoSource
    ├── enc_queue       -> EncodeThread -> stream_queue -> StreamThread
    └── infer_queue     -> InferThread  -> SharedDetections
                                      └-> MqttThread
```

当前对象本身大多已经具备“按实例隔离”的基础：

- `BlockingQueue` 是对象实例；
- `SharedDetections` 是对象实例；
- `InferThread` 自己持有 RKNN context；
- `EncodeThread` 自己持有 MPP context；
- `MqttThread` 自己创建 MQTT client；
- `StreamThread` 自己持有 FFmpeg 输出上下文。

需要重点处理的不是简单复制线程，而是配置归一化、生命周期、路标识、资源争用和输出规则。

## 3. 兼容性原则

### 3.1 单路兼容

现有平面配置继续有效：

```json
{
  "active_config": "config_4k_18mbps.json",
  "source": {},
  "model": {},
  "detection": {},
  "stream": {},
  "mqtt": {}
}
```

程序读取旧配置后，内部自动转换为只有一个元素的路配置列表。单路配置的模型、阈值、跟踪参数、固定 RTMP 地址和日志含义不改变。

### 3.2 多路配置

新增 `channels` 数组。每个元素拥有独立的输入、输出、MQTT 和路级模型覆盖项：

```json
{
  "channels": [
    {
      "id": "channel_a",
      "source": {
        "type": "pull_stream",
        "url": "rtmp://192.168.0.168/live/dual_a",
        "width": 1920,
        "height": 1080,
        "fps": 30
      },
      "stream": {
        "sink": "fixed_rtmp",
        "rtmp_url": "rtmp://192.168.0.168/live/alertgateway"
      },
      "mqtt": {
        "topic": "desk/dual/a",
        "client_id": "AlertGateway-dual-a"
      }
    },
    {
      "id": "channel_b",
      "source": {
        "type": "pull_stream",
        "url": "rtmp://192.168.0.168/live/dual_b",
        "width": 1920,
        "height": 1080,
        "fps": 30
      },
      "stream": {
        "sink": "local_flv",
        "path": "/tmp/alertgateway_dual_b.flv"
      },
      "mqtt": {
        "topic": "desk/dual/b",
        "client_id": "AlertGateway-dual-b"
      }
    }
  ]
}
```

模型和检测参数先采用顶层公共配置，只有确实需要差异时才允许路级覆盖，避免双路配置复制一大段模型参数。

### 3.3 固定输出地址

所有新的或活动的 RTMP 推理结果输出仍必须使用：

```text
rtmp://192.168.0.168/live/alertgateway
```

当前规则下，每个活动推理通道使用固定模板 `rtmp://192.168.0.168/live/alertgateway_{channel_id}`；通道 ID 决定结果地址，禁止使用日期或模式等非通道标识后缀。

如果未来业务要求多路同时提供网络结果流，需要另行确定复用/分发方案；本计划不自行发明新的 RTMP URL 规则。

## 4. 目标代码结构

### 4.1 新增路级流水线

从 `main.cpp` 提取 `ChannelPipeline`，每个实例独立持有：

- `IVideoSource`；
- `enc_queue`、`infer_queue`、`stream_queue`、`mqtt_queue`；
- `InferThread`；
- `EncodeThread`；
- `StreamThread` 或其他输出 sink；
- `MqttThread`；
- `SharedDetections`；
- `channel_id` 和路级统计状态。

目标结构：

```text
main
 ├── normalize_config()
 ├── ChannelPipeline(channel_a)
 │    ├── source / infer / encode / output / mqtt
 │    └── queues / SharedDetections
 └── ChannelPipeline(channel_b)
      ├── source / infer / encode / output / mqtt
      └── queues / SharedDetections
```

`main.cpp` 只负责进程级工作：读取配置、安装信号处理、创建和启动各路、统一停止及返回错误码。

### 4.2 生命周期

启动顺序保持当前原则，每路都先启动下游，再启动上游：

```text
MQTT -> Output -> Encode -> Infer -> Source
```

停止顺序为：

1. 停止所有 Source；
2. 关闭每路输入队列；
3. 停止 Encode 和 Infer；
4. 关闭输出队列；
5. 停止 Output；
6. 关闭 MQTT 队列并停止 MQTT。

任一路启动失败时，必须停止已经启动的其他路，不能留下半启动状态。

### 4.3 路标识

日志和检测消息必须能区分来源：

- 日志增加 `[channel_a]`、`[channel_b]` 前缀；
- `PullStats`、`Infer`、`EncodeStats`、`StreamStats` 均带 `channel_id`；
- MQTT topic 使用独立路级 topic；
- MQTT JSON 增加可选 `channel_id` 字段；
- `Frame` 和 `EncodedPacket` 如需跨路汇总，再增加路标识字段。

当前各路拥有独立队列，因此不能依赖全局帧号判断来源。

## 5. 分阶段实施

### 阶段 0：冻结当前单路基线

目标是证明重构前后单路行为没有变化。

工作内容：

1. 保存当前旧配置单路运行结果；
2. 固定输入、时长、模型、码率和播放器；
3. 记录输入/推理/编码/推流统计；
4. 确认生产 `config/config.json`、V4L2 入口和固定 RTMP 不被改动。

完成条件：旧单路配置有可复查的 180 秒基线。

### 阶段 1：配置归一化

新增 `ChannelConfig` 和配置解析函数：

1. 旧平面配置自动包装成一个 channel；
2. 新 `channels[]` 配置解析为多个 channel；
3. 保留 `active_config` 的兼容合并逻辑；
4. 校验每路必须有唯一 `id`、有效 source 和合法输出 sink；
5. 在启动日志中打印最终路数和每路关键配置。

这一阶段暂不改变线程拓扑，先让单路配置和多路配置都能被正确解析。

### 阶段 2：提取 `ChannelPipeline`

把当前 `main.cpp` 中的单路对象、队列、启动和停止逻辑移动到路级类中。

要求：

- 单路仍只创建一个 pipeline；
- 不复制一套 `main.cpp`；
- 每路 `SharedDetections` 独立，不能共享跟踪状态；
- 每路 MPP、RKNN、FFmpeg、MQTT 资源独立释放；
- 任一路异常能触发全局有序退出。

完成后先只运行旧单路配置，确认指标与阶段 0 基线一致。

### 阶段 3：路级观测和输出 sink

先完善证据，再扩大路数：

1. 为每路统计加 `channel_id`；
2. 将输出从隐含的 `rtmp_url` 抽象为明确 sink；
3. 支持 `fixed_rtmp`、`local_flv`、`metrics_only` 三种验证用途；
4. `fixed_rtmp` 强制校验固定地址；
5. `local_flv` 输出保留媒体信息和文件大小检查；
6. `metrics_only` 不创建第二个网络结果流。

当前双路脚本中使用本地 FLV 的做法可作为过渡验证，但最终应由代码中的明确 sink 类型表达，不能依赖把本地路径填入 `rtmp_url`。

### 阶段 4：双路 1080p PoC

验证顺序固定为：

1. 单独运行 A 路；
2. 单独运行 B 路；
3. A+B 同时运行；
4. 对比三组相同统计窗口。

每路必须核对：

- 输入地址和视频内容是否正确；
- `PullStats`、推理耗时和推理更新数；
- 编码 FPS、码率、`enc_drop`；
- 输出 FPS、`out_drop`、`write_fail`；
- 队列是否持续增长；
- MQTT 消息是否带正确路标识；
- A 路固定 RTMP 是否可观看；
- B 路本地 FLV 是否可解析。

双路合格不代表已经支持任意多路，只代表可以继续研究资源瓶颈。

### 阶段 5：一路 4K + 一路 1080p

只有双路 1080p 稳定后才进行：

- A 路 4K、B 路 1080p；
- 保持相同模型和检测参数；
- 单独记录每路分辨率、码率和耗时；
- 重点观察 NPU、DDR、RGA、MPP、CPU、内存和温度。

暂不直接进行两路 4K，避免在没有资源余量证据时扩大变量。

### 阶段 6：多路扩展与资源调度

如果双路已经暴露 NPU 争用，再决定是否引入共享推理调度器。

当前每个 `InferThread` 都会建立自己的 RKNN context，并请求全部 NPU 核心。多路时可能出现：

- NPU context 争用；
- 推理耗时增加；
- DDR 带宽不足；
- RGA 或 MPP 队列拥塞；
- 温度升高导致降频。

第一版先使用独立 pipeline 实测，不同时引入调度器。只有日志证明 NPU 是首个瓶颈时，才设计共享 RKNN context 或按路调度策略。

## 6. 测试与验收门槛

### 单路门槛

- 旧平面配置和新单元素 `channels[]` 配置都能启动；
- 输出仍为固定 RTMP 地址；
- 输入、编码、输出约 30 FPS；
- `enc_drop=0`、`out_drop=0`、`write_fail=0`；
- 跟踪和 MQTT 行为没有明显回归。

### 双路门槛

- 两个输入流同时 active；
- 两个 channel 的日志、MQTT 和结果数据没有串路；
- 每路错误计数和队列可独立读取；
- A 路固定结果流可观看；
- B 路输出文件或性能数据可验证；
- 能明确指出第一个资源瓶颈，不能只写“两个进程都在运行”。

### 多路门槛

- 逐路增加负载，不直接跳到目标路数；
- 任一路异常时能安全停止全部 pipeline；
- 资源指标和每路统计持续可读；
- 首个饱和点有重复实验支持；
- 未破坏单路、V4L2 生产入口、冻结模型和固定 RTMP 规则。

## 7. 当前不做的事情

- 不通过复制多个 `main.cpp` 分支实现多路；
- 不改变当前生产 V4L2 模型入口；
- 不在多路重构阶段调整检测阈值、跟踪参数或模型；
- 不为每路创建新的结果 RTMP URL；
- 不在双路资源数据不足时承诺三路、四路或两路 4K；
- 不把一次双路运行结果直接写成最终性能结论。

## 8. 完成定义

本计划完成的标准是：

1. 单路旧配置无需修改即可运行；
2. 单路和多路共用同一套路级 pipeline 代码；
3. 双路配置能够独立确认两路数据和资源指标；
4. 输出 sink 和固定 RTMP 约束在代码中有明确校验；
5. 首个多路资源瓶颈能够通过日志和重复实验定位；
6. 所有结论都有对应配置、日志和运行窗口，不依赖肉眼或进程数量判断。

## 10. ROI/Tiling 边界目标处理开发计划（2026-07-20）

### 10.1 问题定义

当一个大目标跨越两个 tile 的边界时，两个 tile 可能分别只识别到目标的左半部分和右半部分，
最终形成两个检测框。当前全局 NMS 主要依赖 IoU；如果两个框只覆盖同一目标的不同部分，IoU
可能很低，NMS 无法合并。

本问题不通过更换输入视频解决，也不立即更换“小目标模型”，优先在 tile 坐标还原后的结果层处理。

### 10.2 阶段一：增加 tile 重叠区域

先把 `tiling.overlap_ratio` 从当前 `0.10` 作为基线，增加候选值：

1. `0.20`；
2. `0.30`；
3. 对比边界大目标的重复框、漏检和推理耗时。

验收要求：

- tile 坐标仍为偶数对齐；
- NV12/YUYV 的源 stride 不变；
- tile 数量不意外增加；
- 单路推理和编码输出不出现花屏、错位或帧率异常。

### 10.3 阶段二：增加跨 tile 边界合并

在每个 tile 检测框还原到整帧坐标后，为检测框增加边界关系判断：

- 记录检测框是否接近 tile 的左、右、上、下边缘；
- 只对相邻 tile、同类别、靠近共同边界的框进入候选合并；
- 判断框之间的水平/垂直间距、中心线重合度和尺寸相似度；
- 满足条件时先做框联合（union），再进入全局 NMS；
- 不满足条件的框保留，避免把两个真实相邻目标误合并。

该阶段必须保留原始检测框日志或调试统计，至少能区分：普通 NMS 合并、边界联合和未合并框。

### 10.4 阶段三：整帧低频补检评估

如果仅靠重叠和边界联合仍无法稳定处理特别大的目标，再评估：

```text
整帧低频推理：负责大目标和全局目标确认
Tile 高频推理：负责小目标和局部细节补充
结果去重：整帧结果与 tile 结果统一坐标后全局 NMS
```

该方案会增加 NPU 负载，必须先测量单路和双路的 NPU 推理耗时、帧率和 `infer_drop`，不能直接作为默认方案。

### 10.5 验证顺序

1. 单路 4K、全画面 ROI、2 tile、重叠率 10% 基线；
2. 单路仅调整重叠率到 20%；
3. 单路启用边界框合并；
4. 对比边界大目标的框数量、漏检率和误合并率；
5. 单路通过后再验证双路 1080p；
6. 最后才评估整帧低频补检。

### 10.6 通过标准

- 跨 tile 的同一大目标最终只保留一个稳定框；
- 两个真实相邻目标不会因靠近边界被错误联合；
- 小目标识别结果不因边界合并明显下降；
- 单路输出保持约 30 FPS，编码/推流无持续丢帧；
- 双路时能明确记录额外 NPU 负载和推理帧覆盖情况；
- 未通过单路验收前，不把边界合并逻辑作为多路默认配置。

### 10.7 当前实现状态

已完成第一版跨 tile 边界合并：

- 每个检测框保留所属 tile 信息；
- 支持左右相邻 tile 和上下相邻 tile；
- 仅合并同类别、贴近相邻边缘、正交方向重叠至少 45%、边界间距不超过 96 像素的框；
- 联合框后继续执行全局 NMS；
- 日志输出 `ROI boundary merges=N`，用于验证实际合并次数；
- 不满足条件的框保持独立，避免简单按距离合并真实相邻目标。

2026-07-20 已完成单路 4K + 2 tile 短时实流验证：日志多次出现 `ROI boundary merges=1`，
说明边界重复框已进入联合逻辑；末尾正常统计约 30 FPS，`enc_drop=0`、`out_drop=0`、
`write_fail=0`，结果流保持正常。尚未完成不同大目标/相邻目标样本下的误合并率验收，下一步应
使用专门样本对比“同一大目标跨边界”和“两个相邻目标靠边界”两类场景。

### 10.8 大框套小框处理

边界整帧复检可能同时产生一个完整大框和一个 tile 小框。仅使用 IoU NMS 时，小框被大框包含，
但交并比可能低于阈值，两个框会同时保留。

已在全局 NMS 增加同类别包含关系抑制：

- 计算交集面积与较小框面积的比例；
- 比例达到 `0.70` 且小框中心位于大框内部时，视为包含重复；
- 置信度接近时优先保留覆盖范围更大的框；
- 不同类别不做包含抑制，保留例如“人”和“手机”的嵌套目标。

已完成单路 ROI 实流验证：整帧复检和 `ROI boundary merges=1` 均持续执行，正常窗口约 30 FPS，
`enc_drop=0`、`out_drop=0`、`write_fail=0`。仍需使用相邻同类目标样本验证不会被错误合并。

### 10.9 前移到 Tracker 之前的联合 ROI 融合

针对手机、笔记本仍偶发重框的问题，处理重点前移到 Tracker 之前：

- 左右/上下 tile 使用 384--512 像素明确重叠，当前 4K ROI 按宽度计算约 448 像素；
- tile 重叠区增加 IoS（交集/较小框面积）去重，同类别重复框优先保留远离 tile 边缘的结果；
- 边界复检改为当前 ROI 复检，并在坐标还原后和 tile 结果联合合并；

### 10.10 公共重叠带联合 ROI 复检

复核发现，旧版本以 tile 的物理外边缘触发复检，无法覆盖两个半框都处于 384--512 像素公共重叠带中间的场景。
现将流程调整为：

1. 从相邻 tile 的同类检测中筛选同时落入公共重叠带、正交方向重合且间距合理的候选对；
2. 以两个候选框的并集加上下文边距生成联合 ROI，单次复检获得完整目标视野；
3. 完整框在进入 Tracker 前替换被其覆盖的 tile 半框；覆盖率达到 50% 或 IoS 达到 35% 即抑制，不能再用中心距离否决半框与完整框；
4. 只在联合 ROI 复检失败时保留原 tile 结果，避免把两个真实相邻同类目标直接合成一个框。

`overlap_ratio` 继续控制最终 384--512 像素 overlap，允许后续在不改代码的情况下做 10%/12%/13% 单变量验证。
