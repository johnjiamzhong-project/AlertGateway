# 检测框消失问题排查记录

## 背景

接入 RGA 硬件加速预处理（见 `docs/RGA预处理加速方案.md`）之后，实际跑起来发现输出视频里一个检测框都没有。排查过程中先后推翻了好几个假设，最终定位到模型量化导出环节，过程比较曲折，完整记录下来，避免以后重复踩坑。

## 排查时间线

### 假设 1：RGA 颜色空间转换出问题 —— 排除

最先怀疑 RGA 把 YUYV→RGB888 时颜色空间不对（`improcess` 的 `usage` 参数默认是 `IM_COLOR_SPACE_DEFAULT`，没显式指定 BT.601）。

验证方法：在 `InferThread.cpp` 里临时把 RGA 转出来的图和原来 CPU 转的图都存成 PPM/PNG，拉下来肉眼对比。两张图几乎一致，颜色没有明显偏差。

尝试显式指定 `IM_YUV_TO_RGB_BT601_LIMIT` 反而导致 `improcess` 直接报错失败（`Failed to call RockChipRga interface`），说明这块板子的 RGA 驱动版本不支持这个组合，改回默认值（不传 color space flag）。

**结论：RGA 图像本身没问题，跟这次的"无框"问题无关。**

### 假设 2：阈值太严 —— 排除

把 `conf_threshold` 从 0.25 调到 0.15，还是一个框都没有。说明不是阈值边界问题。

### 假设 3：类别名字符串没匹配上 —— 排除

检查 `COCO_LABELS` 数组和 `config.json` 里 `target_classes` 的字符串（"cell phone"/"cup"/"keyboard"/"mouse"/"laptop"/"book"），完全匹配，大小写、空格都一致。不是过滤逻辑的字符串比较问题。

### 假设 4（命中一半）：zero-copy 反量化用的 scale/zp 不对

之前 `InferThread.cpp` 用 zero-copy 方式直接拿 NPU 原始 UINT8 输出（`pass_through=0 + RKNN_TENSOR_UINT8`），自己在 `YoloPostprocess::process_dequant()` 里用 `(val-zp)*scale` 反量化。

加诊断打印（追踪整帧最高分对应的 anchor/class/score）发现：**最高分永远是 anchor=0、class=person、score 精确等于 1.0**。每次重启复现，不是随机噪声。

算了一下：`scale=2.60618, zp=-128`，代码里 `dq(v) = (int32_t(v) - zp) * scale`，由于 `v` 是 `uint8_t`（零扩展，永远 ≥0），`v - (-128) = v + 128` 永远 ≥128，乘上 scale=2.6 后永远是几百的大数，sigmoid 后必然饱和到 1.0——不管 v 是多少都一样。

**这一步定位到查询枚举可能用错**：代码用的是 `RKNN_QUERY_OUTPUT_ATTR`，而绑定的内存是 `RKNN_TENSOR_NHWC` 格式的 native 缓冲区，理论上应该用 `RKNN_QUERY_NATIVE_NHWC_OUTPUT_ATTR` 查询匹配的 scale/zp。改了之后**数值没变**（两个查询枚举在这个 SDK/驱动版本下返回的是同一组值），说明这不是查询枚举的问题。

### 假设 5（真正命中）：改用标准浮点输出路径，发现分类通道是精确的 0

放弃手动反量化，改成标准 `rknn_outputs_get(want_float=1)` 让驱动自己处理内部反量化，配合 `YoloPostprocess::process()`（之前写好但从未在生产环境跑过的浮点路径）。

部署后程序运行正常，但**还是没有框**。加诊断打印发现：

```
DEBUG top anchor=0 class=person score=0.5 logit=0
pre_filter_count(>=thresh)=8400   ← 全部 8400 个 anchor 都"达标"
```

`logit` 精确为 0，且**全部 8400 个 anchor 的最高分类 logit 都精确为 0**——这不是统计噪声能解释的，统计上不可能 8400 次独立计算都精确落在 0.0。同时检查输出 buffer：

- `rknn_outputs_get` 返回成功，`size=2822400` 字节，正好等于 `8400 anchor × 84 channel × 4 字节(float32)`，缓冲区大小是对的。
- dump 前几个 float 值（channel-major 排布下对应 anchor0-11 的 cx 坐标）：`7.8, 13.0, 18.2, 20.8, 39.0...`，跟 80×80 网格 stride=8 算出来的像素坐标量级吻合，**说明 box 坐标通道（channel 0-3）的数据是真实有效的，channel-major 索引方式没错**。

box 数据是真的，分类数据精确为 0——索引方式排除了嫌疑，问题缩小到分类通道本身的反量化结果。

中途还测试过"去掉 `RKNN_NPU_CORE_0_1_2` 多核 mask"（怀疑多核拆分破坏了分类分支），结果没有任何变化，排除多核问题。

### 根因：单一共享 scale 把分类精度完全量化抹平

回头看模型导出脚本 `tools/convert_int8.py`，用的是最朴素的 `rknn.build()`，没有指定 `outputs` 参数，整个 84 通道（4 box + 80 class）合并成一个张量，**共享一组 scale/zp**。

参考官方 `rknn_model_zoo` 的 `postprocess.cc` 才意识到：标准做法是把 box 和分类拆成两个独立张量分别量化，因为两者数值量级天差地别：

- box 坐标：像素空间 0-640
- 分类概率：0-1（这个模型的分类分支在 ONNX 图里自带 `Sigmoid` 节点，导出时已经做过一次 sigmoid，输出本来就是概率不是 logit——这也顺带发现了 C++ 代码里重复做了一次 sigmoid 的小 bug）

这套模型量化时按 84 通道整体的数值范围校准出 scale≈2.6，这个 scale 对 box（范围 640）合适，但对分类概率（范围 1）来说，整个 0-1 区间被压缩进不到 1 个 INT8 量化档位——所有概率值无论实际是多少，量化后几乎全部落在同一个格子里，反量化回来自然全部变成同一个值（0）。**分类信息在导出量化这一步就已经被破坏，运行时代码（无论怎么改反量化公式、怎么调阈值）都救不回来。**

## 修复方法

### 1. 模型导出：拆分输出张量

用 `onnx` 库的 `shape_inference` 找到最后一次 `Concat` 之前的两个分支节点：

```python
import onnx
from onnx import shape_inference
m = onnx.load('yolov8s.onnx')
m2 = shape_inference.infer_shapes(m)
# /model.22/Mul_2_output_0    [1, 4, 8400]   box（已解码到像素空间）
# /model.22/Sigmoid_output_0  [1, 80, 8400]  class（已过 Sigmoid）
```

`tools/convert_int8.py` 改为：

```python
rknn.load_onnx(model=ONNX_MODEL,
                outputs=['/model.22/Mul_2_output_0', '/model.22/Sigmoid_output_0'])
```

重新跑量化校准，模型变成 2 个独立输出，各自有自己的 scale/zp。

### 2. C++ 侧适配两个输出

- `InferThread.cpp`：`load_model()` 里按 `n_elems`（4×8400 还是 80×8400）识别哪个 index 是 box、哪个是 class，不依赖固定顺序假设；`run()` 里 `rknn_outputs_get` 同时取两个输出的浮点值，分别传给后处理。
- `YoloPostprocess::process()` 签名改为同时接收 `box` 和 `cls` 两个缓冲区；去掉手动 sigmoid（cls 已经是概率）。

## 验证结果

修复后实测：稳定检测到 3-5 个目标（之前一直是 0），且因为两个输出张量都更小，反量化+后处理耗时（`cnv` 字段）从 ~9ms 降到 ~1.4ms，单帧总耗时反而从 ~44ms 降到 ~33-35ms。

## 教训

1. **怀疑精度问题时，先拿原始数据验证，不要靠调参数试错。** 这次靠的是逐层加诊断打印（最高分、原始 logit、buffer 大小、前几个原始 float 值）才一步步排除掉颜色空间、阈值、字符串匹配、查询枚举、多核 mask 这几个错误方向，最后才定位到真正的根因。如果一开始就盯着调阈值，永远调不出来。
2. **YOLO 类模型导出 INT8 时，box 回归和分类两个分支必须拆成独立输出张量分别量化**，数值量级差异巨大的分支共享一组 scale 会直接摧毁其中精度较敏感的一侧。
3. **不要假设导出的 ONNX 图"应该"是什么样**，实际去读图（`onnx.load` + `shape_inference`）确认节点名和数据流向，比凭经验猜测可靠。
4. 这次"全部 anchor 的最高分都精确等于同一个值"是一个很强的诊断信号——精确相等（不是"接近"）几乎总是意味着量化精度丢失或读取了未初始化/错位的内存，不是真实推理结果的统计噪声。
