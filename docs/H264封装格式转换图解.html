配合 src/stream/StreamThread.cpp 的 parse_annexb / write_extradata / write_packet 看

NALU 是什么

H.264码流不是连续的画面数据，而是一串"网络抽象层单元"(NALU)拼起来的，每个NALU有个type标签，标识自己是画面帧还是配置信息。

SPS/PPS 管什么

SPS(type=7)管序列级参数(分辨率/profile)，PPS(type=8)管图像级参数(熵编码方式等)。画面数据是压缩后的残差，没有这两份配置解码器完全无法解析。

为什么要转换格式

EncodeThread/MPP吐出来的是Annex B(起始码分隔,无长度字段)，但FLV容器要求AVCC(4字节长度前缀,SPS/PPS单独存进extradata只发一次)。
交互演示：一段码流从 Annex B 转成 AVCC
SPS (type 7) PPS (type 8) IDR关键帧 (type 5) 普通帧 (type 1)
① Annex B 原始码流（EncodeThread/MPP 吐出来的格式，起始码分隔）
把鼠标移到任意 NALU 方块上，看它在右边转换成了什么
② extradata：AVCDecoderConfigurationRecord（只在写 header 时打包一次）
SPS/PPS 被从码流里抽走，塞进这个结构，不会再出现在每帧数据里
③ AVCC 帧序列（每帧实际写进 RTMP 包体的内容，不含 SPS/PPS）
每个画面NALU前面换成4字节长度前缀，不再需要扫描起始码找边界
把鼠标移到①里任意一个NALU方块上，看它转换成了②还是③里的哪一段

if (n.type == 7) sps_.assign(n.data, n.data + n.size);
if (n.type == 8) pps_.assign(n.data, n.data + n.size);

// write_extradata()：缓存的SPS/PPS打包成②里那个结构，塞进extradata
extra[p++] = 0x01;                                            // configurationVersion
extra[p++] = sps[1]; extra[p++] = sps[2]; extra[p++] = sps[3]; // profile/compat/level
extra[p++] = 0xFF;  // lengthSizeMinusOne=3，告诉解码器长度前缀是4字节
extra[p++] = 0xE1;  // numOfSPS=1

// write_packet()：剩下的画面NALU(跳过7/8)重新拼成③那样的AVCC格式
avcc.push_back((sz >> 24) & 0xFF); // 4字节大端长度前缀，sz=这个NALU的字节数
avcc.insert(avcc.end(), n.data, n.data + n.size);
