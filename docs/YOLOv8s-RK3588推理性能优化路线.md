# YOLOv8s-RK3588 推理性能优化路线

## 一、背景与目标

AlertGateway 最初使用双输出 INT8 YOLOv8s，在 RK3588S 上锁定 CPU/NPU 性能状态后，
`rknn_run()` 实测约为 **29.4 ms**。当前生产配置已切换为 Rockchip 官方九输出
YOLOv8s INT8，使用 `rockchip_dfl` 后处理；旧双输出模型保留为对照基线。

本轮优化目标：

- 以不明显损失桌面物品检测效果为前提，逐步降低 NPU 推理时间；
- 优先尝试不需要重新训练、容易验证和回滚的方案；
- 中期目标为接近 Rockchip 官方基准，长期目标为进入约 **20 ms** 区间；
- 每项优化必须同时记录延迟和精度，不能只根据单次最低耗时判断效果。

## 当前进度

EXP-001 至 EXP-008 已完成，详细过程见
[`YOLOv8s-RK3588推理优化实验记录.md`](YOLOv8s-RK3588推理优化实验记录.md)。

- 真实AlertGateway 1245帧 `rknn_run` 中位数为29.429 ms，原29.4 ms基线有效；
- 固定CPU6的独立benchmark中，当前模型中位数为29.237 ms；
- `optimization_level=0/1/2/3`中位数为29.249/29.240/29.239/29.237 ms，
  最大差仅0.04%，没有实际性能收益；
- Rockchip官方优化九输出模型在相同CPU6、零拷贝、固定输入和锁频条件下为
  25.662 ms，NPU阶段比双输出基线快约12.2%；
- 当前模型“run+outputs_get+完整后处理”中位数为30.540 ms，官方九输出路径为
  27.746 ms，完整链路净快2.794 ms，约9.15%；
- 官方九输出路径已接入生产 `InferThread`，通过 `model.output_layout` 在
  `decoded` 和 `rockchip_dfl` 之间显式切换；
- 固定图片回归、真实摄像头短跑/长跑、固定桌面场景 A/B、Host Simulator 精度量化
  和板端生产配置 smoke 均已完成；
- 生产配置已切换为 `model.path=model/yolov8s_rockchip_dfl.rknn`、
  `model.output_layout=rockchip_dfl`、`model.conf_threshold=0.25`；
- 输入尺寸 640x480、576x448 和 512x384 候选已在固定标注图片集上评估，均因桌面抽帧
  验证集精度明显退化而不进入生产；
- 下一项为扩充桌面六类数据集并微调，先恢复/提升业务场景精度，再考虑压缩或缩小输入。

Rockchip Model Zoo 当前公布的 RK3588 YOLOv8s INT8 性能为 40.8 FPS，
折算约 **24.5 ms**，并标注为单核测试。因此，29.4 ms 与官方公开基线的实际差距
约为 4.9 ms。网上约 20 ms 的结果可能采用了不同的模型图、输入尺寸、工具链版本、
频率设置或计时口径，不能直接作为同口径结论。

## 二、统一基准

算法优化前先统一测试口径，否则无法判断差异来自模型、运行环境还是测量方法。

### 2.1 对照模型

在同一块板上测试：

1. Rockchip 官方优化版 YOLOv8s RKNN + 官方 Demo；
2. 当前项目的 YOLOv8s RKNN + 独立纯推理程序；
3. 两个模型在同一个 benchmark 程序中的结果；
4. 分别测试单核和 `RKNN_NPU_CORE_0_1_2`。

### 2.2 测试条件

- 固定 Toolkit、RKNN Runtime、NPU Driver 版本；
- CPU、NPU、DDR governor 使用相同设置；
- 记录板卡温度，避免热降频干扰；
- 先预热100次，再测量至少1000次；
- 分别记录 `rknn_run()` 和 `rknn_outputs_get()`；
- 记录平均值、中位数、P90和最大值；
- 不使用单次最低值作为优化结论。

如果官方模型在当前板上也只能达到27～29 ms，应先排查工具链、驱动、频率和测试口径，
而不是直接修改网络结构。

## 三、第一阶段：无需重新训练

### 3.1 测试 RKNN 图优化等级（已完成）

EXP-002 时 `tools/convert_int8.py` 固定使用：

```python
optimization_level=0
```

分别生成 level 0、1、2、3 四个版本，保持ONNX、量化数据集和其他配置不变。

后续脚本已参数化，可通过命令行设置 ONNX、校准目录、输出路径、`optimization_level`
和 `decoded/rockchip_dfl` 输出布局；EXP-002 的结论仍以当时固定其他变量的对照结果为准。

比较内容：

- `rknn_run()` 中位数和P90；
- 模型大小；
- RKNN构建日志中的CPU fallback；
- 六个目标类别的召回情况；
- 是否出现检测框、置信度或坐标异常。

这是成本最低的实验，但不应预期单靠该参数从29.4 ms降到20 ms。

### 3.2 使用 Rockchip 优化检测头（已完成并切换生产配置）

旧 `decoded` 基线模型保留了DFL、框解码和Sigmoid，只在最终Concat前拆分box和class输出。

Rockchip官方优化模型会：

- 从模型内部移除DFL；
- 移除检测头后处理结构；
- 输出三个检测尺度的原始box/class特征；
- 增加置信度求和分支，加快CPU端阈值筛选；
- 在CPU端执行DFL解码和后处理。

官方说明DFL在NPU上的处理性能不佳，将其移到模型外通常可以降低推理时间，
而且不需要重新训练。

实施影响：

- 需要修改 `YoloPostprocess`，支持三个尺度的原始输出；
- 需要在CPU端完成DFL softmax、期望值计算、坐标解码和NMS；
- 必须重新验证INT8量化后的检测精度；
- 单独记录NPU节省时间和CPU后处理增加时间。

该项是第一阶段中最重要的模型图优化。当前已经完成生产接入、精度量化和板端 smoke，
并作为生产默认路径使用；旧 `decoded` 模型保留为对照基线。

### 3.3 输入尺寸改为640×480（首轮可行性验证未通过）

摄像头原始分辨率为640×480，但当前推理输入被拉伸为640×640。

改为固定640×480输入具有以下优势：

- 保留摄像头原始宽高比；
- 避免把480行图像无意义放大到640行；
- 输入像素数量减少25%；
- 理论计算量相应下降；
- 检测Anchor数量由8400降至约6300。

按面积线性粗略估算，29.4 ms可能下降至约22～25 ms。实际收益取决于算子调度、
内存访问和NPU利用率，必须以板端实测为准。

需要同步修改：

- ONNX固定输入尺寸；
- RKNN转换和校准数据（工具脚本已完成参数化准备）；
- 离线精度评估的预处理、输出网格校验和坐标缩放（工具脚本已完成参数化准备）；
- RGA预处理目标尺寸；
- 模型输入尺寸查询和校验；
- Anchor数量及三个检测尺度；
- 检测框坐标缩放逻辑。

当前已完成两个 640x480 候选的 ONNX Runtime 验证、INT8 RKNN 转换和 Simulator 精度评估。
评估使用固定标注图片集，其中桌面部分来自 `ab_test_A.flv` 抽取的 30 帧图片，不是实时摄像头
重新拍摄的现场测试：

- 基于现有 640x640 官方九输出 ONNX 修改静态输入/输出形状得到的候选；
- 由 `ultralytics_yolov8` 定制库 `rk_opt_v1.6` 分支原生导出的 640x480 官方九输出候选。

两者均可转换为全 INT8 RKNN，且离线精度结果完全一致：COCO 子集 mAP 为 68.15%，但
桌面抽帧验证集 mAP 从 640x640 基线的 48.33% 降到 8.33%，laptop Recall 从 96.67%
降到 16.67%。因此当前 640x480 方向的首轮可行性验证未通过，不允许进入生产 C++ 矩形
输入放开或板端实时摄像头 smoke。

后续继续评估 576x448 和 512x384 后，两个更小尺寸同样未通过固定验证集，见下一节。

### 3.4 继续降低输入尺寸（已验证未通过）

640x480 失败后，继续测试了 576x448 和 512x384。按面积估算的理想耗时如下：

| 输入尺寸 | 相对640×640像素量 | 按面积估算的理想耗时 |
|---|---:|---:|
| 640×480 | 75% | 22.1 ms |
| 576×448 | 63% | 18.5 ms |
| 512×384 | 48% | 14.1 ms |

以上只是计算量上限估算，实际耗时不会完全线性缩放。本轮已完成原生 ONNX 导出、INT8
RKNN 转换和 Simulator 固定验证集评估，结果如下：

| 输入尺寸 | COCO Subset mAP@0.5 | 桌面抽帧验证集 mAP@0.5 | 桌面 laptop Recall | 结论 |
|---|---:|---:|---:|---|
| 640x640 基线 | 60.16% | 48.33% | 96.67% | 当前生产可用基线 |
| 576x448 | 61.39% | 13.33% | 26.67% | 失败，不进入生产 |
| 512x384 | 53.61% | 5.00% | 10.00% | 失败，不进入生产 |

因此，当前 YOLOv8s 官方九输出 COCO 权重不再继续做直接输入尺寸缩小实验。该结论来自固定
标注图片集的离线筛选，不等同于小尺寸模型已经完成板端实时摄像头验证；由于离线精度退化
幅度已经足够大，未继续进入板端实拍 smoke。后续应先扩充桌面六类数据并微调，恢复业务
场景精度后，再评估是否继续压缩或缩小输入尺寸。

## 四、第二阶段：需要业务数据和微调

### 4.1 建立桌面场景验证集（已建立初始评估集，仍需扩充）

在修改网络结构前，建立固定验证集，覆盖：

- cell phone；
- cup；
- keyboard；
- mouse；
- laptop；
- book；
- 不同距离、角度、遮挡、光照和背景；
- 空桌面及容易误检的负样本。

至少记录：

- 每类Precision、Recall和mAP；
- 小目标召回率；
- 漏检和误检数量；
- RKNN INT8结果与PyTorch/ONNX结果差异。

当前已有 EXP-005/EXP-006 使用的初始桌面场景30帧 GT 和 COCO 20张子集评估脚本，
可支撑 `decoded` 与 `rockchip_dfl` 的切换决策。后续做输入尺寸、六类微调或轻量模型前，
仍需扩充到更完整的固定验证集，尤其覆盖手机、鼠标、键盘、书本和空桌面负样本。

### 4.2 从80类缩减为6类

当前业务只关心六个COCO类别。训练 `nc=6` 的检测头可以：

- 减少分类头输出通道；
- 减少输出张量和CPU后处理数据量；
- 让模型更专注于桌面业务；
- 为进一步降低输入尺寸或缩小主干提供精度余量。

主干和Neck仍占大部分计算量，因此单独减少类别数不一定能显著降低
`rknn_run()`，主要价值是提高业务精度并支持后续压缩。

### 4.3 构建YOLOv8n与YOLOv8s之间的模型

可设计一个宽度介于YOLOv8n和YOLOv8s之间的模型，例如：

- depth接近YOLOv8s或略微减少；
- width multiplier约为0.375；
- 通道数尽量保持16的倍数；
- 使用YOLOv8s作为预训练权重来源，并针对桌面数据微调。

目标是在精度接近YOLOv8s的同时，使640×480输入下的推理时间进入15～20 ms区间。

### 4.4 按目标尺寸调整检测尺度

先统计真实数据中目标框相对图像的尺寸分布，再决定是否删除某个检测尺度：

- 没有小目标时，可以评估删除P3；
- 没有大目标时，可以评估删除P5；
- 删除P3通常节省更多计算，但最容易损害手机、鼠标等小目标。

该项必须由数据分布决定，不能只凭桌面场景的主观判断。

## 五、第三阶段：结构级优化

### 5.1 逐层算子耗时分析

使用 RKNN profiling 找出：

- NPU耗时最高的卷积或检测头算子；
- SiLU、Sigmoid、Concat等是否单独执行；
- 是否有CPU fallback；
- 通道padding和内存搬运是否异常；
- DFL移出模型前后的热点变化。

只有确认热点后，再决定是否修改激活函数、C2f或Neck。

### 5.2 激活函数与模块替换

可评估：

- SiLU替换为ReLU/ReLU6；
- 减少C2f内部Bottleneck数量；
- 缩小Neck通道；
- 用更轻量且RKNN友好的模块替换C2f；
- 调整通道数，使其满足16或32倍数对齐。

这些修改通常需要重新训练。若profiling没有证明相关算子是热点，不应盲目替换。

### 5.3 结构化通道剪枝

只考虑会改变稠密计算图的结构化剪枝：

- 每轮剪除约10%的低重要度通道；
- 通道数保持16倍数；
- 每轮剪枝后微调；
- 每轮都重新转换RKNN并在板端实测。

不优先考虑普通非结构化稀疏。Rockchip官方AutoSparsity当前注明只支持RK3576，
不能假设RK3588会因此获得推理加速。

### 5.4 知识蒸馏

当自定义轻量模型精度不足时，可使用：

- Teacher：原始YOLOv8s；
- Student：YOLOv8n或自定义0.375宽度模型；
- 蒸馏分类、框回归和中间特征。

知识蒸馏主要用于恢复轻量模型精度，实施成本高于普通微调和结构化剪枝。

## 六、暂不优先的方向

以下方向主要解决量化精度，不是降低同一INT8计算图执行时间的首选：

- 校准算法对比；
- 混合精度量化；
- QAT；
- box分支保留FP16。

这些方案可能提高精度，也可能因引入高精度算子而增加耗时，应放在结构和输入尺寸优化之后。

## 2026-07-11 后处理循环重构与 L1/L2 缓存优化

- **性能分析**：使用带 Debug 符号的 benchmark 和板端 `/usr/lib/linux-tools-5.15.0-185/perf` 对固定正样本输入进行采样。关于 `perf` 工具，此前报告的无法录制问题，实际是因为 `/usr/bin/perf` 包装器版本与内核不匹配，直接调用已安装的 `5.15` 版本路径 `/usr/lib/linux-tools-5.15.0-185/perf` 可正常使用。采样结果确认，在 `rknn_run()` 约 27.5 ms、`outputs_get` 约 1.2 ms 之外，`RockchipYoloPostprocess::process` 占用了 CPU 采样周期的 **66.87%**。其中 NMS 仅占 0.23%。
- **瓶颈定位**：利用 `perf annotate` 定位到 `RockchipYoloPostprocess::process` 的绝对热点在 class 过滤的内层循环中：`score_tensor[class_id * grid_len + cell]` 读取操作。由于原始循环以 `cell` 为外层循环、`class_id` 为内层循环，跨 `class_id` 的内存访问导致了步长为 `grid_len`（最大 6400 字节，远超 cacheline 大小）的非连续跨步访问。此外，由于 `sum_tensor` 对 80 类的 background noise 累加效应，导致 95.4% 的 cells（8016/8400）都通过了 `sum_tensor[cell] < sum_threshold` 过滤，加剧了缓存颠簸（Cache Thrashing）。
- **优化实现**：将 class 过滤循环重构：外层遍历 `class_id`，内层顺序遍历 `cell`。对于 640x640 输入（`grid_len <= 6400`），在栈上预分配 `best_scores` 和 `best_classes` 缓存区（规避堆分配开销）。将跨步访问优化为完全连续的 1-step 顺序访问，最大化 L1/L2 缓存命中率与硬件预取器效率。
- **验证结果**：
  - **精度/一致性**：在固定 desktop 正样本图片上进行 100% 回归校验，检测框、类别、置信度等结果完全一致。
  - **延迟表现**：板端 300 次 benchmark 实测，优化后的 `postprocess` 耗时中位数从 **4.139 ms** 降至 **2.842 ms**，单项净收益为 **1.297 ms**（约 **31.3%** 的后处理加速），完整链路中位数耗时由 **32.833 ms** 降至 **31.597 ms**。
  - **部署状态**：该重构已在源码层合入主线，并成功通过候选二进制验证。目前已正式部署替换为板端正式程序 `~/AlertGateway/AlertGateway`（备份为 `~/AlertGateway/AlertGateway.bak.20260711_205840`），并成功完成了 60 秒 smoke 真机验证，加载、推理、编码及优雅退出全部正常。
