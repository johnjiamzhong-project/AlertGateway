# 第三阶段：设备树入门

> 目标：能看懂设备树，面试能讲，知道UART/I2C/SPI怎么描述

---

## 1. 设备树是什么

### 解决的问题

Linux内核需要支持成千上万种ARM板子。早期做法是每块板子都在内核里写一段硬件描述代码（board file），导致内核里充斥着大量板级代码，难以维护。

设备树（Device Tree）把**硬件描述**从内核代码里分离出来，单独写成一个文件。内核启动时读取这个文件，知道板子上有什么硬件，不需要为每块板子修改内核。

### 完整生命周期

```
.dts 源文件（人类可读文本）
    ↓  dtc 编译
.dtb 二进制文件（/boot/rk-kernel.dtb）
    ↓  uboot 加载，传给内核
内核启动，解析 dtb
    ↓
根据 compatible 匹配驱动
    ↓
根据 status 决定启不启动
    ↓
根据 pinctrl 配置引脚复用
    ↓
设备就绪（/dev/ttyS7 出现）
```

### 文件组织

```
arch/arm64/boot/dts/rockchip/
├── rk3588s.dtsi              # SoC级，描述芯片本身所有硬件（被include）
├── rk3588s-roc-pc.dts        # 板级，描述这块板子用了哪些外设
└── rk3588s-roc-pc-xxx.dts    # 变体，针对不同配置
```

- `.dtsi` 是公共文件，被多个板级文件include
- `.dts` 是最终板级文件，可以覆盖 dtsi 里的默认值
- 板级文件通常只改 `status` 和 `pinctrl`，SoC级节点定义在 dtsi 里

## 2. 四个核心概念

### 2.1 节点（Node）

```dts
serial@feba0000 {
    compatible = "rockchip,rk3588-uart";
    reg = <0x00 0xfeba0000 0x00 0x100>;
    status = "disabled";
};
```

- 每个硬件对应一个节点
- `@` 后面是寄存器基地址，是节点的唯一标识
- 节点可以嵌套（子节点表示从属关系）

### 2.2 属性（Property）

键值对，描述硬件特性：

| 属性 | 作用 |
|---|---|
| `compatible` | 驱动匹配，最重要的属性 |
| `status` | 启用/禁用 |
| `reg` | 寄存器基地址和长度 |
| `clocks` | 时钟依赖 |
| `interrupts` | 中断号 |
| `pinctrl-0` | 引脚复用配置 |

### 2.3 phandle

节点之间的引用，相当于指针：

```dts
// 被引用的节点声明自己的phandle
uart7m2-xfer {
    rockchip,pins = <...>;
    phandle = <0x3f6>;   // 声明自己的phandle
};

// 引用其他节点
serial@feba0000 {
    pinctrl-0 = <0x3f6>; // 通过phandle引用上面的节点
};
```

反编译的dtb里是数字phandle，原始dts源码里是符号引用：
```dts
pinctrl-0 = <&uart7m2_xfer>;  // 源码写法
pinctrl-0 = <0x3f6>;           // 编译后的dtb反编译出来是数字
```

### 2.4 status 取值

```dts
status = "okay";      // 启用
status = "disabled";  // 禁用
status = "reserved";  // 保留给其他用途，不被内核使用
```

## 3. compatible 与驱动匹配

### 匹配原理

内核启动时：
1. 扫描设备树所有节点
2. 每个驱动在代码里定义 `of_match_table`，列出自己能驱动的 compatible 字符串
3. 内核把节点的 compatible 和所有驱动的 `of_match_table` 对比
4. 匹配成功则绑定驱动到节点

### 驱动代码里的 of_match_table（了解即可）

```c
static const struct of_device_id dw8250_of_match[] = {
    { .compatible = "snps,dw-apb-uart" },
    { .compatible = "rockchip,rk3588-uart" },
    { /* sentinel */ }
};

static struct platform_driver dw8250_platform_driver = {
    .driver = {
        .name = "dw-apb-uart",
        .of_match_table = dw8250_of_match,  // 这里
    },
};
```

### compatible 多值写法

```dts
compatible = "rockchip,rk3588-uart\0snps,dw-apb-uart";
```

从左到右匹配，先找更具体的驱动（rockchip专用），找不到再找通用驱动（snps通用）。

## 4. pinctrl：引脚复用

### 为什么需要 pinctrl

SoC的GPIO引脚是复用的，同一个物理引脚可以配置成不同功能：

```
GPIO1_B4 可以是：
  - 普通GPIO输出
  - UART7_RX
  - SPI1_MISO
  - ...
```

pinctrl子系统负责在设备初始化时把引脚切换到正确的功能。

### UART7 的两组引脚方案

```dts
uart7 {
    uart7m1-xfer {
        rockchip,pins = <0x03 0x11 ...>;  // GPIO3_B1/B0，不在20Pin排针上
        phandle = <0x16d>;
    };
    uart7m2-xfer {
        rockchip,pins = <0x01 0x0c ...>;  // GPIO1_B4/B5，对应20Pin排针Pin17/19
        phandle = <0x3f6>;
    };
};
```

改 `pinctrl-0` 就是选择用哪组物理引脚。

### 修改步骤回顾

```bash
# 1. 反编译dtb
dtc -I dtb -O dts -o ~/rk-kernel.dts /boot/rk-kernel.dtb

# 2. 找到目标节点，修改两行
# status = "disabled"  →  status = "okay"
# pinctrl-0 = <0x16d>  →  pinctrl-0 = <0x3f6>

# 3. 编译回dtb
dtc -I dts -O dtb -o ~/rk-kernel-new.dtb ~/rk-kernel.dts 2>/dev/null

# 4. 替换并重启
sudo cp ~/rk-kernel-new.dtb /boot/rk-kernel.dtb
sudo reboot
```

## 5. 三种总线在设备树里的描述

### 5.1 UART（通用异步收发器）

最简单的串行通信协议，两根线TX/RX，点对点通信，不需要时钟线。工业设备、传感器、GPS模块、蓝牙模块大量使用。你做工业上位机时用过的串口就是UART。

适合：低速、简单、点对点、长距离。

```dts
serial@feba0000 {
    compatible = "rockchip,rk3588-uart", "snps,dw-apb-uart";
    reg = <0x00 0xfeba0000 0x00 0x100>;
    interrupts = <GIC_SPI 338 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&cru SCLK_UART7>, <&cru PCLK_UART7>;
    clock-names = "baudclk", "apb_pclk";
    pinctrl-names = "default";
    pinctrl-0 = <&uart7m2_xfer>;
    status = "okay";
};
```

关键点：波特率时钟 `baudclk`、APB总线时钟 `apb_pclk`，两个时钟都要配。

### 5.2 I2C（两线串行总线）

两根线SDA（数据）+SCL（时钟），一个主机可以挂多个从设备，每个从设备有唯一的7bit地址。常见设备：温度传感器、EEPROM、陀螺仪、OLED屏。

适合：短距离、多设备、速度要求不高（100kHz/400kHz）。

```dts
i2c3: i2c@feab0000 {
    compatible = "rockchip,rk3588-i2c";
    reg = <0x00 0xfeab0000 0x00 0x1000>;
    interrupts = <GIC_SPI 332 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&cru CLK_I2C3>, <&cru PCLK_I2C3>;
    clock-names = "i2c", "pclk";
    pinctrl-names = "default";
    pinctrl-0 = <&i2c3m0_xfer>;
    #address-cells = <1>;      // 子节点地址用1个cell表示
    #size-cells = <0>;          // 子节点没有size
    status = "okay";

    // I2C设备挂在总线上，作为子节点
    eeprom@50 {
        compatible = "atmel,24c32";
        reg = <0x50>;           // I2C设备地址
    };
};
```

I2C特有：`#address-cells = <1>` 和 `#size-cells = <0>`，子节点的 `reg` 是I2C地址不是内存地址。

### 5.3 SPI（串行外设接口）

四根线CLK/MOSI/MISO/CS，全双工，速度比I2C快很多。常见设备：Flash存储器、LCD屏、ADC/DAC芯片、以太网模块。

适合：高速、全双工、对速度有要求的场景（几十MHz）。

```dts
spi1: spi@feb40000 {
    compatible = "rockchip,rk3588-spi";
    reg = <0x00 0xfeb40000 0x00 0x1000>;
    interrupts = <GIC_SPI 60 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&cru CLK_SPI1>, <&cru PCLK_SPI1>;
    clock-names = "spiclk", "apb_pclk";
    pinctrl-names = "default";
    pinctrl-0 = <&spi1m2_cs0 &spi1m2_pins>;  // CS和数据引脚分开配置
    #address-cells = <1>;
    #size-cells = <0>;
    status = "okay";

    // SPI设备作为子节点
    flash@0 {
        compatible = "jedec,spi-nor";
        reg = <0>;              // CS编号，CS0
        spi-max-frequency = <50000000>;  // 最高50MHz
    };
};
```

SPI特有：`spi-max-frequency` 限制最高时钟频率，pinctrl通常包含CS和数据线两部分。

### **使用对比：**

|      | 线数 | 速度 | 设备数 | 典型用途           |
| ---- | ---- | ---- | ------ | ------------------ |
| UART | 2    | 低   | 点对点 | 串口调试、工业设备 |
| I2C  | 2    | 中   | 多设备 | 传感器、小屏幕     |
| SPI  | 4    | 高   | 多设备 | Flash、高速传感器  |

### 三种总线细节对比

| | UART | I2C | SPI |
|---|---|---|---|
| 子节点 | 无 | 有（设备地址） | 有（CS编号） |
| 子节点reg含义 | - | I2C地址（7bit） | CS片选编号 |
| 特有属性 | - | `#address-cells=1` `#size-cells=0` | `spi-max-frequency` |
| pinctrl | TX/RX | SDA/SCL | CLK/MOSI/MISO/CS |

## 6. reg 属性详解

```dts
reg = <0x00 0xfeba0000 0x00 0x100>;
```

RK3588S是64位地址，用两个32位cell表示一个地址：
- `<0x00 0xfeba0000>` = 基地址 0xfeba0000
- `<0x00 0x100>` = 长度 0x100（256字节）

内核用这个地址做 `ioremap`，把硬件寄存器映射到内核虚拟地址空间，然后通过读写内存的方式操作硬件。

## 7. Device Tree Overlay

### 是什么

不修改原始dtb，加载一个**增量补丁**（.dtbo文件），只描述变化的部分。

### 为什么用

- 更安全，改错了删掉overlay，原始dtb不受影响
- 支持运行时动态加载/卸载
- 树莓派、Firefly都支持

### 基本用法

```bash
# 加载overlay
mkdir -p /sys/kernel/config/device-tree/overlays/uart7
echo uart7.dtbo > /sys/kernel/config/device-tree/overlays/uart7/path

# 卸载overlay
rmdir /sys/kernel/config/device-tree/overlays/uart7
```

overlay的dts写法：

```dts
/dts-v1/;
/plugin/;

&uart7 {
    status = "okay";
    pinctrl-0 = <&uart7m2_xfer>;
};
```

## 8. 常用调试命令

```bash
# 查看系统识别到的设备树
ls /proc/device-tree/

# 查看某个节点的属性
cat /proc/device-tree/serial@feba0000/status

# 查看aliases（设备别名）
cat /proc/device-tree/aliases/serial7

# 反编译当前运行的dtb
dtc -I dtb -O dts -o ~/current.dts /sys/firmware/fdt

# 查看pinctrl状态（需要root）
sudo cat /sys/kernel/debug/pinctrl/pinctrl-rockchip-pinctrl/pinmux-pins | grep uart

# dmesg查看设备注册信息
dmesg | grep ttyS
dmesg | grep feba0000
```

## 9. 面试常见问题

**Q: 设备树是什么，解决什么问题？**
A: 硬件描述文件，把硬件信息从内核代码里分离出来。内核启动时读取，根据compatible匹配驱动，不需要为每块板子修改内核代码。

**Q: compatible属性的作用？**
A: 驱动匹配。内核把节点的compatible和所有驱动的of_match_table对比，匹配成功则绑定驱动到这个节点。多个值从左到右匹配，先找具体驱动再找通用驱动。

**Q: status有哪些值？**
A: okay（启用）、disabled（禁用）、reserved（保留）。

**Q: pinctrl的作用？**
A: 配置GPIO引脚复用。SoC的物理引脚可以配置成多种功能，pinctrl告诉内核把哪个引脚切换成什么功能。status控制设备启不启动，pinctrl控制信号从哪个物理引脚出来，两个缺一不可。

**Q: dtb是怎么来的，什么时候加载？**
A: dtc编译器把dts源文件编译成dtb二进制，存放在/boot目录。uboot启动时加载dtb，传给内核，内核启动早期解析dtb建立设备树结构。

**Q: 能运行时修改设备树吗？**
A: 不能直接修改运行中的dtb，但可以用Device Tree Overlay动态加载增量补丁，不影响原始dtb。

**Q: I2C设备怎么在设备树里描述？**
A: I2C设备作为控制器节点的子节点，子节点的reg属性是I2C地址（7bit），控制器节点需要设置`#address-cells=<1>`和`#size-cells=<0>`。

**Q: 你有没有实际修改过设备树？**
A: 有，在RK3588S的Firefly板子上启用了UART7。默认status是disabled，pinctrl用的是m1引脚组，不在扩展排针上。我把status改成okay，pinctrl改成m2引脚组（对应20Pin排针的Pin17/Pin19），重新编译dtb替换后重启，/dev/ttyS7就出现了。
